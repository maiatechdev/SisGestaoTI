1. Introdução

Este documento especifica os requisitos de software do SisGestão TI, sistema web destinado à gestão centralizada de chamados de suporte de TI e comunicados internos da SEMDESC — Secretaria Municipal de Desenvolvimento Econômico e Social. O sistema visa substituir o processo manual e descentralizado de registro de chamados, oferecendo rastreabilidade, comunicação em tempo real e controle de acesso baseado em perfis de usuário. O público-alvo são os colaboradores da SEMDESC (usuários finais), a equipe técnica de TI (técnicos) e os gestores (administradores).

2. Escopo do Sistema

Inclui: Autenticação com JWT e controle de acesso por perfil (RBAC); abertura, listagem e acompanhamento de chamados de TI com categorização por prioridade e categoria; chat em tempo real por chamado com persistência de histórico via WebSocket; dashboards personalizados por perfil; módulo de comunicados internos com fluxo de criação, aprovação e publicação.

Exclui: Integração com sistemas externos de monitoramento de infraestrutura; aplicativo mobile nativo; módulo financeiro; integração com Active Directory ou LDAP; exportação de relatórios em PDF ou Excel; gestão de inventário de ativos de TI.

3. Definições, Acrônimos e Abreviações

  • RBAC — Role-Based Access Control (Controle de Acesso Baseado em Perfis)
  • JWT — JSON Web Token (token de autenticação stateless)
  • WS — WebSocket (protocolo de comunicação bidirecional em tempo real)
  • RF — Requisito Funcional
  • RNF — Requisito Não Funcional
  • US — User Story (história de usuário)
  • RB — Regra de Negócio
  • C.I — Comunicado Interno
  • SPA — Single Page Application (aplicação de página única)
  • SEMDESC — Secretaria Municipal de Desenvolvimento Econômico e Social

4. Visão Geral do Produto

O SisGestão TI é uma aplicação web do tipo SPA (Single Page Application), acessada por qualquer navegador moderno sem necessidade de instalação. O sistema é composto por dois módulos principais:

  • Backend REST API + WebSocket: construído em Node.js com Express e TypeScript, responsável pela lógica de negócio, autenticação, persistência em banco SQLite e comunicação em tempo real.
  • Frontend SPA: interface em HTML5, CSS puro e JavaScript vanilla, com roteador hash e sidebar de navegação adaptada ao perfil do usuário logado.

O acesso é controlado por três perfis: user (colaborador), ti (técnico de suporte) e admin (gestor/administrador), cada um com permissões e visualizações específicas.

5. Requisitos Funcionais

RF-001: O sistema deve permitir o login com e-mail e senha, retornando um token JWT com validade de 8 horas.
RF-002: O sistema deve diferenciar três perfis de acesso: usuário (user), técnico de TI (ti) e administrador (admin).
RF-003: O sistema deve permitir que qualquer usuário autenticado abra um chamado informando título, descrição, prioridade e categoria.
RF-004: O sistema deve exibir ao usuário comum apenas seus próprios chamados; técnicos e administradores visualizam todos.
RF-005: O sistema deve permitir filtrar chamados por status, prioridade e categoria via parâmetros de consulta.
RF-006: O sistema deve permitir que técnicos e administradores atualizem o status de um chamado (aberto, em andamento, resolvido, fechado).
RF-007: O sistema deve permitir que técnicos e administradores atribuam e removam a atribuição de chamados a si mesmos.
RF-008: O sistema deve disponibilizar um chat em tempo real por chamado via WebSocket, autenticado por token JWT.
RF-009: O sistema deve exibir o histórico completo de mensagens ao conectar ao chat de um chamado.
RF-010: O sistema deve persistir todas as mensagens de chat no banco de dados.
RF-011: O sistema deve permitir que técnicos e administradores criem comunicados internos.
RF-012: Comunicados criados por técnicos devem ficar com status "pendente" até aprovação de um administrador.
RF-013: Comunicados criados por administradores devem ser publicados imediatamente com status "aprovado".
RF-014: O sistema deve permitir que administradores aprovem ou rejeitem comunicados com status "pendente".
RF-015: O sistema deve exibir dashboards personalizados de acordo com o perfil do usuário autenticado.
RF-016: Administradores devem poder excluir comunicados internos.

6. Requisitos Não Funcionais

RNF-001: O sistema deve ser acessível via navegador web moderno sem instalação de software adicional.
RNF-002: As senhas devem ser armazenadas com hash bcrypt, fator de custo mínimo 10.
RNF-003: O token JWT deve expirar em 8 horas e ser validado em todas as requisições às rotas protegidas.
RNF-004: O sistema deve responder a requisições REST em menos de 500ms em condições normais de operação.
RNF-005: A interface deve ser funcional em resoluções de tela a partir de 1024×768 pixels.
RNF-006: O código do backend deve ser escrito em TypeScript para garantia de tipagem estática.
RNF-007: O banco de dados não deve depender de instalação de servidor externo (SQLite via sql.js/WASM).
RNF-008: A conexão WebSocket deve ser autenticada por token JWT na URL de conexão.
RNF-009: O sistema deve impedir acesso a recursos de outros usuários por meio de validação de propriedade no backend.

7. User Stories

US-001: Como usuário, eu quero abrir um chamado de TI para registrar formalmente um problema e acompanhar sua resolução.
US-002: Como usuário, eu quero enviar mensagens no chat do meu chamado para me comunicar diretamente com o técnico responsável.
US-003: Como usuário, eu quero ver o dashboard com o status dos meus chamados para ter uma visão rápida do andamento dos meus atendimentos.
US-004: Como técnico de TI, eu quero visualizar todos os chamados abertos para priorizar e organizar os atendimentos da equipe.
US-005: Como técnico de TI, eu quero atribuir chamados a mim mesmo para sinalizar que estou trabalhando no atendimento.
US-006: Como técnico de TI, eu quero atualizar o status de um chamado para manter o solicitante informado sobre o progresso.
US-007: Como técnico de TI, eu quero criar comunicados internos para informar os usuários sobre manutenções programadas ou avisos importantes.
US-008: Como administrador, eu quero aprovar ou rejeitar comunicados criados por técnicos para garantir a qualidade das informações publicadas.
US-009: Como administrador, eu quero visualizar um dashboard com métricas gerais do sistema para monitorar a performance do suporte de TI.
US-010: Como administrador, eu quero ver chamados críticos em destaque no dashboard para priorizar ações imediatas.

8. Casos de Uso

UC-001 — Gerenciar Chamados (Técnico/Admin): O técnico ou administrador acessa a lista de chamados, filtra por status ou prioridade, seleciona um chamado, atualiza seu status, se auto-atribui e interage via chat em tempo real.
UC-002 — Abrir Chamado (Usuário): O usuário preenche o formulário com título, descrição, prioridade e categoria; o sistema cria o chamado com status "aberto" e redireciona para a tela de detalhe.
UC-003 — Publicar Comunicado (Admin): O administrador cria um comunicado que é publicado imediatamente; técnicos criam comunicados que entram em fila de aprovação.
UC-004 — Aprovar/Rejeitar Comunicado (Admin): O administrador visualiza comunicados pendentes no dashboard ou na lista de comunicados e os aprova ou rejeita com um clique.

9. Regras de Negócio

RB-001: Apenas técnicos de TI e administradores podem atualizar o status de chamados; usuários comuns não têm esta permissão.
RB-002: Um usuário comum somente acessa seus próprios chamados; tentativas de acessar chamados de outros usuários retornam erro 403.
RB-003: Comunicados criados por técnicos iniciam com status "pendente" e só ficam visíveis a todos os usuários após aprovação de um administrador.
RB-004: Um comunicado aprovado ou rejeitado não pode ter seu status alterado novamente; o fluxo é unidirecional (pendente → aprovado | rejeitado).
RB-005: O token JWT deve ser incluído em todas as requisições às rotas protegidas no header Authorization como Bearer token.
RB-006: A conexão WebSocket de chat exige token JWT válido e ID de ticket acessível pelo usuário; conexões inválidas são encerradas com código 4001 ou 4003.
RB-007: Chamados recém-criados recebem automaticamente o status "aberto" e o usuário criador como solicitante.

10. Protótipos de Interface

O sistema foi desenvolvido com interface funcional em HTML/CSS puro. As telas implementadas incluem: tela de login, dashboard adaptativo (3 versões por perfil), lista de chamados com filtros, formulário de abertura de chamado, detalhe do chamado com chat em tempo real, lista de comunicados com ações de aprovação/rejeição e formulário de criação de comunicado. O repositório do projeto está disponível em: https://github.com/maiatechdev/SisGestaoTI

11. Critérios de Aceitação

RF-001 será aceito quando o login retornar token JWT válido para credenciais corretas e erro 401 para credenciais inválidas.
RF-004 será aceito quando usuário comum listar apenas seus próprios chamados e técnicos/admin visualizarem todos.
RF-008 será aceito quando dois clientes conectados ao mesmo chamado trocarem mensagens em tempo real, com persistência confirmada via GET do detalhe do chamado.
RF-012/013 serão aceitos quando comunicado de técnico iniciar como "pendente" e comunicado de admin iniciar como "aprovado" automaticamente.
